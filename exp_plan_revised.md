# 1\. 实验目标与研究问题

本计划将 Group E 重构为两个版本：E0 为原始拼接版 baseline，E1 为关系增强版。验证"当前包的即时状态"是否能够在历史行为统计之外提供额外判别信息，以及这种增益是否必须通过关系建模才能体现。

- E0 是否仅仅复现了 Group B 的能力，还是能在部分设备上提供额外帮助。
- E1 加入关系特征后，是否能够形成"当前状态 + 历史基线 + 偏离程度"的真正互补。
- 哪些设备类型更可能从 E1 中受益，例如低流量设备、切换频繁设备、移动终端等。
- 阶段化冷启动处理是否能降低早期样本统计不稳定对结果的影响。

# 2\. E0 与 E1 的定义

| **版本** | **核心结构**                                                        | **作用**                            | **是否保留** |
| -------- | ------------------------------------------------------------------- | ----------------------------------- | ------------ |
| E0       | 当前包单包特征 + 历史窗口统计 + 冷启动补偿                          | 作为 baseline，验证直接拼接是否有效 | 保留原设计   |
| E1       | 当前包即时状态子集 + 历史行为基线子集 + 关系特征 + 阶段化冷启动特征 | 验证关系增强后是否真正形成互补      | 新增改进版   |

# 3\. 样本构造规则

## 3.1 当前包选择（方案 C）

对每个参考时间点，只在其之前的有限时间范围内搜索当前包，候选区间定义为：

\[t_ref - D_max, t_ref\]

其中 D_max 表示当前包允许距离参考时间点的最大回溯距离。

在该区间内，当前包按以下规则选择：

- 优先选择距离 t_ref 最近且满足 payload_len > 0 的包；
- 若不存在有效荷载包，则退回选择距离 t_ref 最近的任意包；
- 若区间内没有任何包，则该样本丢弃。

若参考时间点对应的最近包本身满足 payload_len > 0，则可直接将其作为当前包。

当前包确定后，其时间记为 t_cur，历史窗口定义为：

(t_cur - W, t_cur)

其中 W 为历史窗口大小。历史行为统计仅使用该区间内、且严格早于当前包的流量计算，不包含当前包本身。

此外，记录辅助特征：

anchor_gap = t_ref - t_cur

用于表示当前包距离参考时间点的实际偏差，帮助模型判断该包的"当前性"强弱。

在实验中固定 D_max，例如设为 5s或者3s，但是考虑到会和采样步长S=5耦合，或许3s是更好的选择，或者更改步长的选值。

## 3.2 历史窗口

历史窗口继续采用仅使用过去信息的时间窗口，范围为 (t−W, t)，不允许使用当前包及未来流量，避免信息泄露。历史统计特征仍遵循 Group B 的定义，但统计对象限定为当前包之前的历史流量。

## 3.3 冷启动阶段化处理

对于历史不足 W 秒的样本，不丢弃，而是按照历史完整比例 hist_window_ratio 分为三个阶段：cold（<0.3）、partial（0.3-0.8）、full（≥0.8）。同时保留连续可靠性特征，如 hist_duration_actual、hist_window_ratio、hist_packet_count_actual。阶段特征帮助模型区分"几乎没有历史""部分历史"和"完整历史"三种统计可信度不同的状态。

# 4\. E1 的特征融合原则

- 不按来源简单拼接，而按角色组织：当前包即时状态、历史行为基线、当前包相对历史的关系特征、历史可靠性特征。
- 优先保留能表达 A 和 B 单独都不容易表达的信息，尤其是"当前包是否符合最近历史主模式"。
- 控制冗余：E1 不追求把 A 和 B 全量堆叠，而是保留对互补最有帮助的子集。
- 尽量减少对环境强相关字段的依赖，避免 E1 完全被端口、TTL、tcp_window 等实现/环境指纹主导。

# 5\. E1 特征设计说明

E1 由四类特征组成。下面给出建议纳入的特征、其作用以及选取理由。为保持实现可控，建议先采用"小而强"的特征集合，而不是一次性加入大量复杂交互项。

## 5.1 当前包即时状态特征（来自 Group A 的保留子集）

| **特征**             | **作用**                     | **为什么选它**                                                 | **备注**                                   |
| -------------------- | ---------------------------- | -------------------------------------------------------------- | ------------------------------------------ |
| packet_len           | 表示当前包总长度             | 直接反映当前包的消息规模；很多设备在关键状态下包长分布不同     | 用于与历史均值/方差形成偏离关系            |
| payload_len          | 表示当前包有效载荷长度       | 比总长度更贴近业务内容大小；能区分空载/控制包与实际数据包      | 适合与历史 payload 统计比较                |
| payload_entropy      | 表示当前包载荷复杂度/随机性  | 可反映压缩、加密或报文结构差异；在同长度条件下提供额外判别信息 | 对摄像头、控制型设备、移动终端均可能有帮助 |
| delta_t              | 表示当前包与前一包的时间间隔 | 表达当前时刻的局部节律，适合识别突发、心跳、切换点             | 后续可与历史 IAT 统计结合                  |
| protocol / tcp flags | 表示当前包所属协议及控制状态 | 体现当前通信状态，例如控制握手、确认、数据传输等               | 推荐编码为少量稳定字段                     |
| direction            | 表示当前包方向               | 单独判别力不一定很强，但与历史主方向结合时有价值               | 主要用于关系特征，不建议单独依赖           |

## 5.2 历史行为基线特征（来自 Group B 的保留子集）

| **特征**                                     | **作用**                     | **为什么选它**                                               | **备注**                                     |
| -------------------------------------------- | ---------------------------- | ------------------------------------------------------------ | -------------------------------------------- |
| packet_rate / byte_rate                      | 刻画历史窗口内发包和字节速率 | 形成当前状态的动态背景，判断设备近期是否处于活跃或低活跃状态 | 是最重要的历史基线之一                       |
| iat_mean / iat_std / iat_max                 | 刻画历史窗口内到达间隔统计   | 描述节律稳定性与突发性                                       | 便于与当前 delta_t 构造关系特征              |
| packet_len_mean / min / max / std            | 描述历史窗口的包长度分布     | 为当前包长度提供参照基线                                     | 是 E1 中关系特征的基础                       |
| payload_len_mean / std                       | 描述历史窗口载荷长度分布     | 帮助判断当前包是否为典型数据包或异常包                       | 与 payload_len、entropy 配合使用             |
| unique_peer_ip_count                         | 描述历史窗口内通信对象多样性 | 固定功能设备通常对象更集中，移动设备更可能变化大             | 对区分专用设备与通用终端有帮助               |
| https_port_ratio / well_known_dst_port_ratio | 描述服务使用和协议偏好       | 反映业务模式和服务依赖范围                                   | 作为历史行为概括保留，不建议过度扩展端口细节 |
| tcp / udp / icmp count or ratio              | 描述历史窗口协议组成         | 不同设备的协议使用习惯差异明显                               | 用于定义历史主模式                           |

## 5.3 关系特征（E1 核心新增）

| **关系特征**                                      | **作用**                                 | **为什么选它**                                   | **实现建议**                        |
| ------------------------------------------------- | ---------------------------------------- | ------------------------------------------------ | ----------------------------------- |
| curr_packet_len_minus_hist_mean / ratio / zscore  | 表示当前包长度相对历史包长分布的位置     | 回答"当前包是典型包还是偏离历史的包"             | 至少保留差值和 z-score 两种形式     |
| curr_payload_len_minus_hist_mean / ratio / zscore | 表示当前载荷相对历史载荷分布的位置       | 帮助识别当前是否出现突发数据或异常轻量控制包     | 与 payload_entropy 联合使用效果更稳 |
| curr_iat_minus_hist_iat_mean / ratio / zscore     | 表示当前时延是否偏离历史节律             | 能识别突发、切换、心跳到达等动态行为             | 使用历史 iat_mean/std 计算          |
| curr_protocol_match_hist_major                    | 表示当前协议是否与历史主导协议一致       | 用一个布尔量表达"当前是否延续主模式"             | 可由历史窗口内占比最高协议定义      |
| curr_direction_match_hist_major                   | 表示当前方向是否与历史主导方向一致       | 对于上下行不平衡设备尤其有意义                   | 建议二值化                          |
| curr_dst_port_in_hist_topk                        | 表示当前目的端口是否属于历史常见端口集合 | 表达当前服务是否延续既有模式，避免只看绝对端口值 | top-k 建议取 3 或 5                 |

## 5.4 历史可靠性与阶段特征

| **特征**                 | **作用**                     | **为什么选它**                                         | **备注**              |
| ------------------------ | ---------------------------- | ------------------------------------------------------ | --------------------- |
| hist_duration_actual     | 真实可用历史时长             | 区分"统计来自 3 秒历史"还是"统计来自完整 30/60 秒历史" | 连续量                |
| hist_window_ratio        | 真实历史时长 ÷ 目标窗口时长  | 直接表示历史完整程度                                   | 连续量                |
| hist_packet_count_actual | 历史窗口内实际包数           | 统计稳定性高度依赖样本数                               | 连续量                |
| hist_stage               | cold / partial / full 三阶段 | 把不同历史可信度显式离散化，帮助模型理解阶段差异       | 可 one-hot 或整数编码 |

# 6\. 明确不作为 E1 主体依赖的特征

以下字段并非完全禁止使用，但不建议作为 E1 的主要互补来源：src_port、dst_port 原始值、ip_ttl、tcp_window 等。这些字段在同数据集内可能很强，但更容易携带环境或实现指纹。E1 的目标是增强"当前状态相对历史"的表达，而不是重新回到绝对指纹主导。

# 7\. 数据处理与切分规范

- 继续使用时间顺序 70 / 15 / 15 的训练、验证、测试切分。
- 必须先切分 packet-level 数据，再在各子集中独立构造 E0/E1 样本，避免历史特征跨集合引用造成信息泄露。
- 当前包与历史窗口的计算只能使用各自子集内部数据。
- 若窗口末端附近没有高信息包，可回退为最近普通包，但必须记录选择规则以保证复现。

# 8\. 实验矩阵

实验聚焦于：直接拼接是否有效，以及关系增强后是否形成互补。

| **阶段**   | **版本**        | **说明**            | **建议参数**                                              | **输出**            |
| ---------- | --------------- | ------------------- | --------------------------------------------------------- | ------------------- |
| E-Baseline | E0              | 原始拼接版 baseline | 沿用现有 S、W 搜索或固定一组中等参数                      | metrics_e0.CSV      |
| E-Improve  | E1              | 关系增强版          | 建议先固定 S=10s, W=30s 做主对比；若有效再做小范围搜索    | metrics_e1.CSV      |

# 9\. 调优与评估建议

- E0 与 E1 的使用一样的参数构建。
- 评估指标使用沿用之前的F1等指标
- 需单独观察 Android Phone 2、Samsung Galaxy Tab、TP-Link Smart plug 等难设备，判断 E1 是否在这些类别上真正带来增益。

# 10\. 输出产物与论文可用结论

- sensitivity_group_e_e1.csv：E1 小范围参数搜索结果（若执行）
- metrics_group_e0.csv / metrics_group_e1.csv：最终评估指标
- overall_summary.csv：与 A/B/D 的统一对比结果
- feature_importance_group_e1.csv：E1 重要性排序，用于解释哪些关系特征真正起作用

若 E1 优于 E0，可在论文中得出：即时特征与历史统计并非天然互补，必须通过关系建模显式表达"当前状态相对历史基线的偏离或延续"，互补效应才更容易显现。若 E1 仍未超过 B，则说明历史行为统计仍是主导信息，而当前包的增益有限，这同样是一个有价值的负结果。

**备注：本计划将 E0 作为稳定 baseline 保留不变，E1 作为本轮唯一正式改进版本。**
